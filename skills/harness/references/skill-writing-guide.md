## 1. Writing Effective Descriptions

The `description` field is the skill's **only trigger mechanism**. Claude sees only `name + description` from the `available_skills` list when deciding whether to invoke a skill.

### How Triggering Works

Claude tends not to invoke skills for tasks it can easily handle with its default tools. Simple requests like "read this PDF" may not trigger a skill even with a perfect description. Complex, multi-step, specialist tasks trigger more reliably.

### Authoring Principles

1. Describe **what the skill does** AND **specific trigger situations**
2. Include boundary conditions that distinguish this skill from similar-but-should-not-trigger cases
3. Be slightly "pushy" — compensate for Claude's conservative triggering tendency

### Good Examples

```yaml
description: "Performs all PDF operations: reading, text/table extraction, merging,
  splitting, rotating, watermarking, encryption/decryption, OCR, and more. ALWAYS
  use this skill when the user mentions a .pdf file or requests a PDF output.
  Especially useful when transformation, editing, or analysis is needed — not just
  simple reading."
```

```yaml
description: "Handles all spreadsheet operations on Excel/CSV/TSV files: adding columns,
  formula calculation, formatting, charts, and data cleaning. Use this skill whenever
  the user mentions a spreadsheet file — even casually ('the xlsx in my downloads')."
```

### Bad Examples

- `"A skill for processing data"` — too vague; no file type or action specified
- `"PDF-related tasks"` — no action list; no trigger situation described

## 2. Body Writing Style

### Why-First Principle

LLMs make better decisions on edge cases when they understand the reason, not just the rule. Context beats coercion.

**Bad:**
```markdown
ALWAYS use pdfplumber for table extraction. NEVER use PyPDF2 for tables.
```

**Good:**
```markdown
Use pdfplumber for table extraction. PyPDF2 is optimized for text extraction and
does not preserve row/column structure in tables. pdfplumber recognizes cell boundaries
and returns structured data.
```

### Generalization Principle

When a problem is found in feedback or testing, fix it at the **principle level** — not as a narrow patch for the specific failing case.

**Overfitting fix:**
```markdown
If the column is named "Q4 Revenue", convert it to a number.
```

**Generalized fix:**
```markdown
If a column name contains keywords implying a numeric value (e.g., "revenue", "amount",
"quantity"), convert that column to a numeric type. If conversion fails, keep the
original value.
```

### Imperative Voice

Write instructions in imperative form ("do X", "use Y"), not hedged form ("you can do X", "it is possible to Y"). Skills are directives, not suggestions.

### Context Economy

Every sentence costs tokens. Ask before writing:
- "Does Claude already know this?" → delete it
- "Will Claude make a mistake without this?" → keep it
- "Would one concrete example communicate this better than a paragraph?" → use the example

## 3. Defining Output Formats

Use this pattern when the output structure matters:

```markdown
## Report Structure

Follow this template exactly:

# [Title]
## Summary
## Key Findings
## Recommendations
```

Keep format definitions concise. A concrete example is more effective than a long description.

## 4. Writing Examples

Examples outperform long explanations:

```markdown
## Commit Message Format

**Example 1:**
Input:  Add JWT-based user authentication
Output: feat(auth): implement JWT-based authentication

**Example 2:**
Input:  Fix the password visibility toggle button not working on the login page
Output: fix(login): repair password visibility toggle button
```

## 5. Progressive Disclosure Patterns

### Pattern 1: Domain Separation

```
bigquery-skill/
├── SKILL.md             (overview + domain selection guide)
└── references/
    ├── finance.md        (revenue, billing metrics)
    ├── sales.md          (opportunities, pipeline)
    └── product.md        (API usage, features)
```

When the user asks about revenue, load only `finance.md`.

### Pattern 2: Conditional Detail

```markdown
# DOCX Processing

## Document Creation
Create new documents with docx-js. See [DOCX-JS.md](references/docx-js.md).

## Document Editing
Simple edits: modify the XML directly.
**If tracked changes are needed:** see [REDLINING.md](references/redlining.md)
```

### Pattern 3: Large Reference File Structure

Reference files over 300 lines must include a table of contents at the top:

```markdown
# API Reference

## Table of Contents
1. [Authentication](#authentication)
2. [Endpoint List](#endpoint-list)
3. [Error Codes](#error-codes)
4. [Rate Limits](#rate-limits)

---

## Authentication
...
```

## 6. Script Bundling Criteria

Observe agent transcripts during test runs. Bundle into `scripts/` when you see:

| Signal | Action |
|--------|--------|
| Same helper script generated in 3 out of 3 tests | Bundle in `scripts/` |
| Same `pip install` / `npm install` run every time | Add dependency install step to skill |
| Same multi-step approach repeated every run | Describe as standard procedure in skill body |
| Same error followed by the same workaround every time | Document known issue + fix in skill |

All bundled scripts must pass an execution test before being committed.

## 7. Data Schema Standards

Use these standard schemas for consistent data exchange between skills.

### eval_metadata.json

Metadata for each test case:

```json
{
  "eval_id": 0,
  "eval_name": "descriptive-name-here",
  "prompt": "The user's task prompt",
  "assertions": [
    "The output contains X",
    "A file was created in format Y"
  ]
}
```

### grading.json

Assertion-based scoring result:

```json
{
  "expectations": [
    {
      "text": "Output contains 'Seoul'",
      "passed": true,
      "evidence": "Step 3 output: 'extracting Seoul region data'"
    }
  ],
  "summary": {
    "passed": 2,
    "failed": 1,
    "total": 3,
    "pass_rate": 0.67
  }
}
```

**Field name rules:** Use exactly `text`, `passed`, `evidence` — do NOT use variants like `name`/`met`/`details`.

### timing.json

Execution time and token measurement:

```json
{
  "total_tokens": 84852,
  "duration_ms": 23332,
  "total_duration_seconds": 23.3
}
```

Save `total_tokens` and `duration_ms` immediately from the sub-agent completion notification. This data is only accessible at notification time and cannot be recovered afterward.

## 8. What NOT to Include in a Skill

- `README.md`, `CHANGELOG.md`, `INSTALLATION_GUIDE.md`, or other supporting documents
- Meta-information about the skill generation process (test results, iteration history)
- User-facing explanatory docs — skills are directives for AI agents, not documentation for humans
- General knowledge Claude already has

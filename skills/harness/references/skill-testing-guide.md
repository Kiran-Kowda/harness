# Skill Testing & Iterative Improvement Guide

Methodology for validating and improving skills generated by harness. Supplementary reference for SKILL.md Phase 6.

---

## Testing Philosophy

Combine **qualitative** and **quantitative** evaluation:

| Evaluation type | When to use | Method |
|----------------|-------------|--------|
| Qualitative | Subjective quality: style, design, tone | Direct human review |
| Quantitative | Objective outcomes: file creation, data extraction | Assertion-based auto-scoring |

---

## Writing Test Prompts

Avoid abstract instructions. Use **specific, natural-language prompts** that mirror real user requests.

**Bad (too abstract):**
```
"Process a PDF"
```

**Good (specific and natural):**
```
"In 'Q4_Revenue_Final_v2.xlsx' in my Downloads folder, add a profit margin column
using columns C and D, then sort by it."
```

Good test prompts:
- Reference real-looking file names
- Include context (folder, format, goal)
- Sound like something a real user would type

---

## Comparative Evaluation

For each test, run **two versions in parallel**:

1. **With skill** — Claude uses the skill
2. **Without skill** — Claude handles the task with default tools

Compare results to measure the skill's real value. If the skill version isn't meaningfully better, the skill needs work.

---

## Assertion-Based Scoring

Define objectively verifiable assertions for each test:

```
"Was a profit margin column added?"
"Is the file sorted in descending order?"
"Were no existing columns removed?"
```

Score each assertion as PASS or FAIL. Record evidence for each.

Use the `grading.json` schema from `skill-writing-guide.md` for consistent output.

---

## Trigger Validation

For every skill, define two lists:

### Should-trigger prompts (3+ examples)
Prompts where this skill SHOULD be invoked:
```
"Extract the table from report.pdf"
"Convert pages 3-5 of invoice.pdf to images"
"Merge these three PDFs into one"
```

### Should-NOT-trigger prompts (3+ examples)
Prompts where this skill should NOT be invoked (but could be confused):
```
"Summarize this article" (no PDF involved)
"Read this text file"
"What does this image show?"
```

Test both lists. If a should-trigger prompt doesn't fire the skill, strengthen the description. If a should-NOT-trigger prompt fires it, add boundary conditions to the description.

## Iterative Improvement Process

### Step 1: Collect feedback
- User reports after real use
- Patterns in `_workspace/` artifacts
- Failed assertions from scoring

### Step 2: Identify the root cause
Ask: is this a skill quality issue, a role gap in an agent, or a workflow sequencing issue?

| Root cause | Fix target |
|-----------|-----------|
| Skill gives wrong instructions | Skill body |
| Agent missing capability or role coverage | Agent definition |
| Wrong order of operations | Orchestrator |

### Step 3: Generalize the fix
Never patch for the specific failing case only. Fix at the principle level (see `skill-writing-guide.md` §2 Generalization Principle).

### Step 4: Remove unnecessary work
If a step or instruction is never useful, remove it. Smaller skill bodies = less token cost + more reliable triggering.

---

## Using Specialist Agents for Testing

For complex skills, use dedicated test agents:

- **`skill-tester`** — runs test prompts and records results
- **`grader`** — evaluates assertions and writes `grading.json`
- **`skill-improver`** — reads grading results and proposes skill edits

This mirrors harness's own design: skills for tasks, agents for roles.

---

## Workspace Organization

Store all test artifacts in `_workspace/` with consistent naming:

```
_workspace/
├── eval_metadata.json
├── test-01/
│   ├── prompt.md
│   ├── output/
│   └── grading.json
├── test-02/
│   └── ...
└── timing.json
```

Preserve `_workspace/` between runs — it enables incremental re-testing and documents the harness's evolution over time.
